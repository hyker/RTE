The client verifies the CRL's digital signature against Intel's certificate to ensure it has not been tampered with.

Server verification

This validates the quote received from the remote RTE server to verify the RTE's authenticity and integrity.

Expected ID: —

The verification confirms two things:

  • Authenticity: that the server is running inside a genuine Intel TDX trusted execution environment.
  • Integrity: that the code running inside it has not been tampered with.

Authenticity is guaranteed by a digital signature from a private key unique to the specific CPU, authenticated by Intel.

Integrity is guaranteed by comparing the unique ID in the remote server's quote with the expected ID embedded in this client.

Important note: The expected ID is embedded in this client and is critical to guarantee the RTE server is legitimate. If you do not trust the server that you loaded this client from, then you should obtain the expected ID from a trusted channel and compare it with the value shown above.

Technical note: As server ID, we use the RTMR2 value contained in the TDX quote. RTMR2 measures the full boot chain (kernel, initramfs, bootloader configuration), which includes the dm-verity root hash of the filesystem. This means the RTMR2 value transitively covers the entire service — both the boot chain and all application code — ensuring the ID is cryptographically bound to every component running inside the enclave.

TOE File (Target of Evaluation)
Your file will be encrypted locally on your device before upload. It can only be decrypted inside the verified trusted enclave — no one else can access it, not even us.
Encryption is performed in your browser using a public key extracted from the server's attested TDX quote. The corresponding private key exists only inside the enclave and is never exposed. This ensures that your data remains confidential in transit and at rest, and can only be processed within the isolated environment whose integrity was verified in the previous step.
